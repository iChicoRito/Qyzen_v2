# Implementation Guide: Building the Student Assessment-Taking Experience

This is a build blueprint for the student assessment-taking experience described in the companion walkthrough. It explains **what to build, in what order, and the rules each part must follow** — in plain language, without code. Think of it as a checklist and set of requirements a team could hand to a developer.

**Starting point:** this guide begins *after* the student has started (entered) the assessment — from the moment the assessment screen loads with their questions, through to finishing and reviewing results. Everything before that point is a precondition (see below), not part of this build.

Each section lists: what the piece is, what it must do, and the traps to watch for.

---

## Precondition (handled before this guide)

Before the assessment screen ever loads, the system has already decided the student is allowed to take it: signed in as a student in good standing, enrolled in the subject and section, inside the schedule (or individually granted access), the assessment has questions, and an attempt is still available. If any of that fails, the student is redirected away and never reaches the screen this guide describes.

This guide assumes that gate has passed. One rule from it still matters here, though, and is repeated at submission time: **anything that affects a student's score or their allowance to take the assessment must be decided by the server, not the student's browser.** The browser can display and assist, but must never be trusted to enforce the rules — a determined student can tamper with anything running on their own device.

---

## How to Approach the Build

Build in the order below. Each phase produces something you can see and test before moving on, and later phases depend on earlier ones:

1. The assessment screen shell (the focused layout)
2. Showing questions and capturing answers
3. View modes and navigation
4. The timer
5. Auto-saving and resuming
6. Integrity rules (anti-cheat)
7. Submitting and grading
8. The results screen
9. Retakes

---

## Phase 1 — The Assessment Screen Shell

**What it is:** The stripped-down, distraction-free layout the questions live inside — the first thing the student sees once they've started.

**What it must do:**
- Hide the app's normal side navigation for the duration of the assessment.
- Show a header area with the assessment's details (code, subject, section, term, educator, schedule), the short instructions, a Save Progress control, and a toggle for the two view modes.
- Reserve a floating spot (top corner) for the timer that sits above the questions.
- Adapt to screen size: spread out on large screens, stack and shrink on phones.

**Watch out for:**
- Pop-up messages (warnings, submit confirmation, time's-up) should overlay the whole screen and dim what's behind, so they can't be ignored.

---

## Phase 2 — Showing Questions and Capturing Answers

**What it is:** Rendering each question and recording what the student chooses or types.

**What it must do:**
- Support two question types:
  - **Multiple choice** — show the options as selectable cards labeled A–D; the student picks exactly one, and the selection is visually highlighted.
  - **Identification** — show a text box for a short typed answer.
- Lay multiple-choice options out in two columns on wide screens and a single column on narrow ones.
- Keep every answer in one place as the student works, keyed to its question, so it can be saved, restored, and submitted as a set.
- If the educator enabled shuffling, randomize the question order *and* the multiple-choice option order — but do this **once per attempt** and keep it fixed for the rest of that attempt, even across a page refresh.
- If the educator enabled hints, surface them at randomized moments during the session.

**Watch out for:**
- Never send the correct answers to the student's browser while the assessment is in progress. The browser should have no way to reveal the answer key.
- The shuffle must be stable within an attempt — a student who refreshes must see the same order, or they'll lose their place and their saved answers won't line up.

---

## Phase 3 — View Modes and Navigation

**What it is:** The two ways to move through the questions.

**What it must do:**
- **List view:** all questions stacked on one scrolling page, with a Submit control at the bottom.
- **Slideshow view:** one question at a time, with Previous and Next controls and a "Question X of Y" counter. Disable Previous on the first question, and replace Next with Submit on the last.
- Let the student switch between the two modes at any time without losing answers.

**Deliberately not built (keep it this way unless asked):**
- No flag/bookmark-a-question-for-later feature.
- No clickable map that jumps directly to an arbitrary question.

**Watch out for:**
- Switching view modes must preserve all answers already entered.

---

## Phase 4 — The Timer

**What it is:** The visible countdown that enforces the time limit.

**What it must do:**
- Show remaining time in hours:minutes:seconds.
- Change color as time runs low (green → yellow → red) and draw attention (a shake) in the final stretch.
- Also display how many integrity warnings the student has left.
- Let the student drag it to a different position and hide/show it.
- When time reaches zero, automatically submit the assessment and show a "Time's Up!" message before moving to results.

**Watch out for:**
- The remaining time must be based on when the attempt actually started, not on a fresh countdown each time the page loads. If a student refreshes, the timer must resume from the real elapsed time — the clock keeps running while they're away.
- The browser's countdown is for display; the server should also know the true start time so time can't be extended by tampering.

---

## Phase 5 — Auto-Saving and Resuming

**What it is:** Keeping the student's work safe as they go, and letting them pick up where they left off.

**What it must do:**
- Save answers automatically a short moment after the student stops typing or selecting (no button press needed).
- Only save when something actually changed, to avoid needless saves.
- Offer a manual Save Progress button as well, which confirms with a brief "saved" message.
- Store each in-progress attempt so that on a refresh, closed tab, or brief disconnect, the student returns to their saved answers with the timer continuing from where it was.
- Keep only one in-progress attempt per student per assessment.

**Watch out for:**
- A failed save (e.g., a momentary connection drop) must **not** interrupt the student — log it quietly, let them keep working, and let the next save catch up.
- Resuming applies only to unfinished attempts. A submitted attempt is final and can't be reopened.

---

## Phase 6 — Integrity Rules (Anti-Cheat)

**What it is:** The set of restrictions and detectors that keep the assessment fair. From the student's side, each triggers a warning; enough warnings ends the attempt.

**What it must do — restrictions:**
- Block copy, cut, paste, and right-click within the assessment.
- Block common inspect/developer keyboard shortcuts, and detect when developer tools are opened.
- Briefly blank the screen if the Print Screen key is pressed.

**What it must do — detections (each counts as a warning):**
- Switching away: changing browser tab, minimizing, or clicking to another app/window.
- Moving the mouse completely off the page.
- Making the window too small, or arranging it side-by-side with another window (split-screen).

**How warnings behave:**
- Each detected violation raises a warning message naming what happened and how many warnings remain, and quietly records the updated count.
- When the count reaches the educator-set limit, the assessment is submitted **immediately and automatically** with whatever answers exist at that moment.
- Ignore rapid duplicate triggers — count a given violation at most once within a short cooldown so a single event doesn't rack up several warnings.
- On touch devices, skip the checks that don't apply (things a phone or tablet can't do), so students aren't punished for the impossible.

**Watch out for:**
- Treat these as fairness aids, not guarantees — they discourage and catch casual cheating but can't be perfect. Keep the final decision (grading, attempt limits) on the server regardless.
- The running warning count must survive a refresh, so it should be saved alongside the answers.

---

## Phase 7 — Submitting and Grading

**What it is:** Turning in the assessment and calculating the score.

**What it must do:**
- On Submit, show a confirmation step:
  - If questions are unanswered, warn with the count and offer a "Submit Anyway" choice as well as a way back.
  - If everything is answered, remind the student the score is final.
- Grade on the server:
  - **Multiple choice:** compare the student's pick to the correct option, ignoring differences in case and spacing.
  - **Identification:** accept a match against any of the accepted correct answers, ignoring case and spacing.
- Decide pass/fail against the passing mark (75%).
- Record the finished attempt with its score, pass/fail status, submission time, and warning count.
- Send the student to their results afterward.

**Watch out for:**
- Grading happens only on the server, using the answer key the browser never saw. Re-verify the access and attempt rules from the precondition at submission time — don't assume they still hold just because the screen loaded earlier.
- Automatic submissions (time up, or warning limit reached) go through the same grading path as a manual submit.
- Notifying the educator that a student submitted is a nice-to-have; if that step fails, it must not fail the student's submission.

---

## Phase 8 — The Results Screen

**What it is:** The summary the student sees after finishing.

**What it must do:**
- Arrange as two panels on wide screens (a narrower summary on the left, wider details on the right) that stack on small screens.
- **Summary panel:** a score dial (percentage, correct vs. incorrect), the raw score, the best score across all attempts, pass/fail status, educator name, submission time, time limit, whether shuffle was on, warnings used, and the schedule.
- **Details panel:**
  - An attempt history listing every submitted attempt with score and pass/fail.
  - A question-by-question review. If the educator allowed review, show the correct answers next to the student's answers with clear right/wrong coloring; if not, show only the student's own answers and whether each was correct — never the answer key.

**Watch out for:**
- Only the owner of a result may view it — check this on the server.
- Respect the "allow review" setting strictly: when review is off, the correct answers must not appear anywhere on the results screen.

---

## Phase 9 — Retakes

**What it is:** Letting a student try again when permitted.

**What it must do:**
- Show a Retake control only when attempts remain — counting both the retakes the educator allows for everyone and any extra attempts granted to that student individually.
- On retake, start a fresh attempt (new blank session for the same assessment).
- When no retakes remain, show a Back to Assessments control instead.

**Watch out for:**
- "Attempts remaining" must be recalculated from finished attempts versus the total allowed each time — don't rely on a stale count.
- Confirm the retake is still allowed at the moment the new attempt starts, not just when the button was shown.

---

## Cross-Cutting Reminders

- **Server is the source of truth** for timing, integrity outcomes that end an attempt, grading, and retake limits (and the entry checks in the precondition). The browser assists and displays.
- **Never expose the answer key** to the browser during an attempt, and only reveal it on results when review is explicitly allowed.
- **Fail gently on the student's side:** connection hiccups and background saves should never block a student mid-assessment.
- **Design for phones too:** layouts stack, and integrity checks skip what doesn't apply to touch devices.

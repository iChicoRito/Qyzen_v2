# Objective
## Verify Background Automation, Preserve Uploaded Assets, and Improve Notification Reliability

---

## Role

You are a Laravel and DevOps engineer with expertise in scheduled tasks, cron jobs, file storage, background processing, and notification systems. Approach this objective by ensuring production reliability, data integrity, and maintainable system behavior.

---

## Description

Perform a comprehensive review of the application's background automation, scheduled processes, notification lifecycle, and file persistence mechanisms to ensure they function correctly in the production environment hosted on Hostinger. Validate that all existing cron jobs and scheduled tasks execute as intended, identify any implementation issues, and correct them where necessary. Ensure uploaded announcement images persist across application updates and deployments without being lost. Improve the notification system by verifying automated cleanup, adding missing administrative notifications, and refining the notification user interface behavior for consistency.

---

## Primary Objective

Audit, correct, and optimize the application's scheduled automation, file persistence, and notification features to ensure reliable production behavior.

---

## Secondary Objectives

- Verify that the automated database backup process executes correctly through Laravel scheduling and Hostinger cron jobs.
- Review and correct the existing cron job configuration if necessary, and provide the appropriate production cron commands.
- Ensure educator announcement images remain permanently available after new code deployments or Git updates.
- Verify that notifications for both educators and students are automatically deleted after three days as intended.
- Add an administrative notification whenever a student successfully confirms their email address.
- Modify the notification interface so that the **Mark All as Read** action remains visible after being used.

---

## Success Criteria

- The database backup process executes successfully through the scheduled automation.
- All cron jobs are verified and corrected where necessary.
- Updated and correct Hostinger cron commands are provided.
- Announcement images remain intact after every deployment or Git update.
- Notifications for educators and students are automatically deleted after three days.
- Administrators receive a notification whenever a student confirms their email.
- The **Mark All as Read** button remains visible alongside the **Delete All** button after being clicked.

---

## Context & Dependencies

- The application is deployed on Hostinger.
- The current production cron jobs are:

  - `domains/qyzen.space/public_html/artisan queue:work --stop-when-empty --max-time=55`
  - `domains/qyzen.space/public_html/artisan schedule:run >> /dev/null 2>&1`

- The project uses Laravel Artisan commands for queue processing and task scheduling.
- The announcement feature currently stores uploaded images that are lost after Git deployments, indicating an issue with file persistence or deployment strategy.
- Notification cleanup relies on an automated scheduled process.

## Completed Work Summary

| Area | Before | After |
|---|---|---|
| Saved copies of school records | A saved copy could appear complete even when some information was missing. It could also be lost during a website update. | Every saved copy now includes all available information or clearly stops when something goes wrong. Saved copies are kept in a safe place that is not replaced during website updates. |
| Automatic daily work | The instructions used by the website host were incomplete, so automatic work could fail without being noticed. | The daily work now has clear and correct instructions. Saved copies and old-notification cleanup are checked every day. |
| Announcement pictures | Pictures added by educators could disappear after a website update. | New pictures are kept in a safe location. Existing pictures can be moved safely, and older pictures remain viewable while the move is being completed. |
| Old notifications | Only some read educator notifications were removed. Student notifications and unread notifications could remain for a long time. | Read and unread notifications for educators and students are removed once they are at least three days old. Administrator notifications are kept. |
| Student email confirmation | Administrators were not informed when a student confirmed an email address. | Active administrators receive one notification when a student confirms an email address for the first time. Opening the confirmation link again does not create another notification. |
| Notification actions | **Mark All as Read** disappeared when there were no unread notifications or after it was selected. | **Mark All as Read** and **Delete All** remain visible at all times. |

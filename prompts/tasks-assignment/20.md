# Objective
## Prevent Warning-Limit Submission and Refresh Bypass During Student Quizzes

---

## Role

You are a full-stack web application developer specializing in secure online assessment workflows, server-side validation, and anti-cheating enforcement.

---

## Description

Fix the vulnerability that allows students to bypass automatic quiz submission after consuming all permitted cheating warnings. Currently, when the final warning triggers automatic submission, a student may refresh the page at the same time and interrupt the submission process, allowing them to continue answering despite reaching the maximum warning limit, such as **3/3 warnings**. Enforce the warning limit through persistent, server-validated quiz state so refreshing, closing, reconnecting, or reopening the page cannot restore access to the active quiz. Once the maximum warning count is reached, the attempt must be permanently locked and submitted without relying solely on the student's browser.

---

## Primary Objective

Permanently lock and submit a student's quiz attempt immediately after the maximum cheating-warning limit is reached, preventing refreshes or interrupted client-side requests from bypassing enforcement.

---

## Secondary Objectives

- Persist every warning count so it cannot be reset or lost when the page refreshes.
- Prevent students from continuing to answer once the configured warning limit has been reached.
- Ensure interrupted automatic submission is completed or enforced when the student reloads or reopens the quiz.
- Preserve the previously entered answers available at the moment the quiz attempt becomes locked.

---

## Success Criteria

- Each cheating warning is recorded persistently and validated by the server.
- Reaching the final allowed warning, such as **3/3**, immediately changes the quiz attempt to a locked or forced-submission state.
- The server rejects answer updates after the maximum warning limit has been reached.
- Refreshing during automatic submission does not return the attempt to an active or answerable state.
- Reopening the quiz after reaching the warning limit displays the attempt as submitted, locked, or already terminated.
- A failed, cancelled, delayed, or interrupted client-side submission cannot bypass the warning limit.
- Repeated warning events or submission requests do not create duplicate submissions or inconsistent attempt records.
- The final warning enforcement works correctly on both desktop and mobile devices.

---

## Constraints

- Students must not be allowed to continue answering after consuming all permitted warnings.
- Enforcement must not rely exclusively on browser-side state, JavaScript execution, page navigation, or a single submission request.
- Refreshing, closing, reopening, or reconnecting to the quiz must not reset the warning count or unlock the attempt.
- Legitimate fullscreen transitions introduced by the quiz workflow must not be incorrectly counted as cheating warnings.

---

## Context & Dependencies

- The student quiz page already includes warning alerts and a maximum warning-attempt limit.
- The current implementation attempts to submit the quiz automatically after the final warning.
- A race condition exists when the student refreshes the page while the automatic submission is occurring.
- The solution must integrate with the existing quiz-attempt records, answer-saving process, submission workflow, warning detection, and fullscreen behavior.

## Detailed Breakdown

### Authoritative Server-Side Warning State

Treat the server-stored warning count and quiz-attempt status as the authoritative state. Client-side counters may be used for display purposes, but they must never determine whether a student is still permitted to answer. Every warning event must be validated and recorded by the server before the interface reflects the updated count.

### Atomic Final-Warning Enforcement

When the final allowed warning is recorded, update the warning count and change the attempt to a locked or forced-submission state within one protected server-side operation. This transition must occur before the browser receives the response so that refreshing or cancelling the request cannot leave the attempt active.

### Answer Protection After Locking

Before saving an answer, verify that the quiz attempt remains active and below the warning limit. Once the attempt is locked, submitted, or terminated, reject all subsequent answer changes, including requests sent from stale tabs or previously loaded pages.

### Interrupted Submission Recovery

Do not depend on a single browser-triggered submission request to enforce the final warning. When the student reloads or reopens the quiz, retrieve the attempt state from the server. When the maximum warning limit has already been reached, finalize any incomplete forced submission or keep the attempt permanently locked instead of allowing the quiz to resume.

### Idempotent Submission

Ensure the forced-submission process can be requested multiple times without creating duplicate records, changing the final result incorrectly, or reopening the attempt. Simultaneous warning, answer-saving, refresh, and submission requests must resolve to one consistent final state.

### Preserving Existing Answers

When the warning limit is reached, retain the latest answers already accepted by the server. Lock the attempt immediately and use the last successfully persisted answer state for the forced submission so the refresh bypass cannot provide additional answering time.

# Task 18 ‚Äî Student-Facing Production Error Audit & Fix

Summary of the audit and fixes for `prompts/tasks-assignment/18.md`: the "403 | This action is
unauthorized" error during exams, the dead hint buttons (math minigame included), and the
auto-retake loop after submission.

---

## Plain-language summary (for non-technical readers)

Three things were reported broken for students taking a quiz, and all three turned out to be
real, reproducible bugs ‚Äî not one-off glitches. Here's what was actually happening and what
changed:

**1. The "403" error mid-exam.** This page popped up when a teacher changed a student's
enrollment status while that student was still taking the quiz (for example, editing or removing
them from a class roster). The security check that causes this is working correctly ‚Äî a student
who's no longer enrolled shouldn't be able to keep answering ‚Äî but the *page shown* was a blank,
dead-end technical error screen with no explanation and no way back, and there was no record
anywhere of why it happened. Now, if this happens, the student is sent back to their assessment
list with a plain message ("You no longer have access to this assessment"), and the system quietly
logs exactly who, what assessment, and when ‚Äî so support staff can actually investigate instead of
guessing.

**2. The hint button (and the math mini-game) doing nothing when clicked.** The hint feature
opens a small game the student has to win to earn a hint. The button worked fine in testing but
could silently stop working in production with zero error message ‚Äî a component the button
depends on (a popup/modal library) wasn't guaranteed to be ready the instant the page finished
loading, and the original code just gave up silently if that happened, leaving the button looking
clickable but doing nothing. It's now fixed to wait briefly for that component to be ready, and if
it genuinely never loads, the button visibly disables itself with an explanation instead of
pretending to work. Also fixed: if a student's answer wasn't saving in the background for any
reason, they previously got zero warning ‚Äî now that failure is recorded so it can be diagnosed.

**3. Getting pushed into a second attempt right after finishing.** This could happen if a
student's final submission got sent twice ‚Äî for example, spotty school Wi-Fi drops the
confirmation the server sent back, the student sees an error and tries submitting again, or their
browser reloads the page at just the wrong moment. Previously, a second submission arriving after
the first one had already been graded would get treated as a brand-new attempt ‚Äî silently using up
one of the student's allowed retakes and sending their teacher a duplicate "a student submitted"
notification. Now the system recognizes "this attempt was already graded" and just shows the
student their existing result instead of creating a confusing extra attempt.

All three fixes were verified by actually logging in as a real student account in the app and
reproducing each scenario, not just by reading the code.

## Technical summary

Investigation used three parallel Explore agents (one per reported symptom) to trace each bug to
its exact root cause in the codebase before any change was made. All three were confirmed as real
defects with concrete file:line locations ‚Äî not phantom/stale reports (the project's own
`PARITY_AUDIT.md`, which pre-dates the hint mini-game feature, incorrectly still says hints were
dropped; the actual feature was added later in `feat(student): task 01/02`, 2026-07-09).

### 1. 403 mid-exam ‚Äî presentation + observability gap, not a policy bug

**Root cause:** `AssessmentPolicy::view()` (`app/Policies/AssessmentPolicy.php:17-28`) correctly
gates student access via `tbl_enrolled.is_active` (`Assessment::scopeVisibleTo()`). It's re-checked
on every quiz request (`take`/`draft`/`hint`/`submit`) ‚Äî that's correct-as-designed enrollment
gating (a CLAUDE.md security invariant) and was **not** changed. The actual defects:
- `take()` (GET) was the only endpoint where a failed `authorize()` rendered Laravel's raw,
  dead-end default 403 HTML page.
- No logging existed anywhere on this failure path, matching the task's own complaint that
  production errors were unreadable.
- `saveDraft()`'s autosave `fetch()` never checked `response.ok` ‚Äî a 403/422/429 during autosave
  was silently discarded, so answers could stop persisting with zero feedback.

**Fix ‚Äî `app/Http/Controllers/Student/QuizController.php`:**
- Added `private function authorizeQuizAccess(Assessment $assessment): void` (lines 40-53) ‚Äî wraps
  `$this->authorize('view', $assessment)`, and on `AuthorizationException` logs
  `Log::warning('Student quiz access denied', ['student_id', 'assessment_id', 'route'])` before
  rethrowing. All four actions (`take`, `saveDraft`, `hint`, `submit`) now call this helper instead
  of the bare `authorize()` call.
- `take()` additionally catches the rethrown exception and redirects to
  `student.assessments.index` with a flash message, instead of letting the exception bubble to the
  framework's default 403 view. `saveDraft()`/`hint()`/`submit()` already returned JSON gracefully
  on this exception via Laravel's `expectsJson()` path ‚Äî no behavior change needed there beyond the
  new logging.
- `resources/views/student/take-quiz.blade.php` ‚Äî autosave's `save()` function now checks `r.ok`;
  on failure it `console.error`s the status/body and keeps `dirty = true` so the next answer change
  retries (previously only network-level failures retried; HTTP error statuses were dropped
  silently).

**Verified live:** logged in as a seeded student, deactivated their enrollment mid-attempt via
`tinker`, reloaded `take-quiz` ‚Äî redirected cleanly to the assessment list (no raw 403 page).
`storage/logs/laravel.log` recorded:
`[WARNING] Student quiz access denied {"student_id":1660,"assessment_id":153,"route":"student.take-quiz"}`.
Repeated with an in-progress attempt open and answered a question ‚Äî console showed
`[quiz] autosave failed 403 ...` instead of silence.

### 2. Hint buttons / math minigame ‚Äî silent client-side failure guard

**Root cause:** `take-quiz.blade.php:327` (pre-fix): `if (!modalEl || typeof KTModal === 'undefined') return;`
silently exited the entire hint-handling IIFE with **zero diagnostics** whenever `KTModal` (the
KTUI modal component) wasn't yet defined at the moment this inline script ran. When tripped, the
`.qz-hint-btn` click-listener binding block never executed ‚Äî buttons rendered but did nothing, no
console error, matching the report exactly. This is a single shared failure point for all 5
mini-games (math included); `renderMath()` itself was verified bug-free in isolation. The backend
(`routes/web.php:76-77`, `QuizController::hint()`, `QuizGradingService::revealHint()`) was already
correctly wired and covered by passing tests.

**Fix ‚Äî `resources/views/student/take-quiz.blade.php:317-437`:**
- `if (!modalEl) return;` now only short-circuits when hints are genuinely off for this session
  (expected ‚Äî no button exists in that case either).
- Button binding was extracted into `bindHintButtons()` and moved behind a
  `waitForKTModal(attempts)` poll (up to ~2s, 20 √ó 100ms) that retries until `KTModal` is defined.
- If it's still undefined after the retry window, `console.error('[quiz] KTModal did not load ‚Äî hint buttons disabled')`
  is logged and every `.qz-hint-btn` is set `disabled = true` with an explanatory `title` tooltip ‚Äî
  turning a silent dead-end into an observable, honest failure state.
- No change to `renderMath`/other game renderers or the intentional 8-second `showIntro` countdown
  before each mini-game ‚Äî both verified correct and working as designed per `prompts/tasks/02.md`.

**Verified live:** as a seeded student on a hint-enabled assessment (`hint_count=5`), clicked Hint
repeatedly until the Math Challenge appeared, solved `17 - 6 = 11`, the correct answer ("Lasso
Tool") was revealed inline and the modal closed ‚Äî zero console errors, both hint POSTs returned
`200 OK`.

### 3. Auto-retake loop ‚Äî grading race condition, not an unconditional loop

**Root cause:** under normal config there's no automatic loop ‚Äî retaking requires an explicit
button click gated by `can_take` (`scores/show.blade.php:277-298`). The real gap:
`QuizGradingService::grade()` (`app/Services/QuizGradingService.php:66-124`, pre-fix) looked up the
in-progress attempt via `Score::firstOrNew(['status' => 'in_progress', ...])` with **no row lock
and no idempotency check**. If a submit response was lost (flaky network) and the client let the
student resubmit, or a back/forward-cache page reload raced an in-flight submit, the **second**
`grade()` call would find no in-progress row (the first had already flipped it to `passed`/
`failed`) and would create a **brand-new** `Score` ‚Äî drawing a fresh question pool, silently
consuming a retake slot, and firing a second `quiz_submitted` notification to the educator.

**Fix ‚Äî `app/Services/QuizGradingService.php`, `grade()` (already inside `DB::transaction()`):**
- The attempt lookup now uses
  `Score::where('student_id', ...)->where('assessment_id', ...)->where('status', 'in_progress')->lockForUpdate()->first()`
  ‚Äî serializes concurrent submits against the same row.
- If no in-progress row is found, the service now checks for an already-graded (`passed`/
  `failed`) row for this student+assessment and **returns it directly**, skipping re-grading,
  re-notifying, and re-touching `StudentAssessmentAccess` ‚Äî closing the duplicate-attempt race and,
  as a side effect, preventing the duplicate educator notification. A brand-new `Score` is only
  constructed as a defensive fallback when truly none exists.
- No schema migration required.

**Verified:** added regression test
`test_grade_is_idempotent_when_called_twice_for_the_same_attempt` in
`tests/Feature/Student/StudentFeaturesTest.php` ‚Äî calls `grade()` twice for the same
student/assessment and asserts the second call returns the same `Score` id and only one row exists
in the DB. Passing.

### Proactive audit

Checked all remaining `authorize()` calls in student controllers
(`AnnouncementController`, `ChatController`, `MaterialController`) and all `fetch()` usage across
`resources/views/student/**`. No other student view uses an un-checked `fetch()` ‚Äî the pattern was
isolated to `take-quiz.blade.php`. A few other student GET routes share the same "raw 403 page on
authorize failure" shape, but they're far lower-traffic than the exam flow and outside this task's
critical scope (active exam sessions) ‚Äî flagged, not changed.

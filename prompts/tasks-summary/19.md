# Task 19 - Mandatory Fullscreen Quiz Mode

Summary of the implementation for `prompts/tasks-assignment/19.md`: students must acknowledge a
fullscreen requirement before interacting with the quiz, fullscreen is requested when the browser
allows it, and the existing anti-cheat warnings no longer count the app's own fullscreen
transitions as cheating.

---

## Plain-language summary (for non-technical readers)

Students now see a required message before a quiz starts. It explains that fullscreen mode is used
to reduce accidental cursor exits and false cheating warnings. The quiz stays blocked until the
student clicks **I Understand**.

After the student acknowledges the message, the browser is asked to enter fullscreen. If the
browser supports fullscreen, the student is kept in that mode during the assessment; if they leave
fullscreen before submitting, the page asks them to return. If the browser cannot use fullscreen
reliably, such as in some mobile or embedded browser environments, the quiz still continues after
acknowledgment instead of unfairly blocking the student or adding a cheating warning.

The fullscreen dialog uses the shared modal component, cannot be closed with the normal close
button or Escape key, and avoids the small internal scrollbar seen during UI review.

Submission still works the same way: once the quiz is successfully submitted, the page exits
fullscreen if needed and sends the student to the result page. The old copy/paste, tab switch,
resize, focus-loss, and cursor-exit warnings still exist; they now ignore only the short browser
events caused by the fullscreen workflow itself.

## Technical summary

No controller, route, schema, or public response contract changed.

### 1. Required fullscreen acknowledgment

**Fix - `resources/views/student/take-quiz.blade.php`:**
- Added a required pre-quiz dialog using the shared `<x-modal>` component, with the heading
  **Fullscreen mode is required**, explanatory copy, and a single **I Understand** button.
- Added `id="qz-session"` around the quiz grid so the content can be marked `inert` before
  acknowledgment on browsers that support the native `inert` property.
- The modal remains open until acknowledgment and swallows Escape before the quiz starts.

**Fix - `resources/views/components/modal.blade.php`:**
- Added three optional props to the shared modal component: `persistent`, `showClose`, and
  `scrollable`.
- The fullscreen dialog uses those props to remove the close button, keep the modal persistent,
  and render a non-scroll body.

### 2. Fullscreen lifecycle

**Fix - `resources/views/student/take-quiz.blade.php`:**
- Added `fullscreenElement()`, `requestQuizFullscreen()`, `exitQuizFullscreen()`, and
  `isFullscreenSupported()`.
- `requestQuizFullscreen()` uses the native Fullscreen API and vendor-prefixed fallbacks.
- Unsupported or rejected fullscreen requests unlock the quiz after acknowledgment for mobile
  compatibility.
- `fullscreenchange` and prefixed listeners call `restoreFullscreen()` when a supported browser
  exits fullscreen during an active quiz.
- The app-controlled fullscreen transition guard re-checks fullscreen state when it ends, so a
  missed browser event still shows the return-to-fullscreen prompt.
- Successful JSON submit calls `exitQuizFullscreen()` before redirecting to the result page.

### 3. Anti-cheat compatibility

**Fix - `resources/views/student/take-quiz.blade.php`:**
- Added `authorizedFullscreenTransition`, a short guard around app-controlled fullscreen entry and
  exit.
- The shared `violation()` function now returns early when the quiz has not started yet or when an
  authorized fullscreen transition is in progress.
- Existing warning behavior remains otherwise unchanged.

### Regression coverage

**Fix - `tests/Feature/Student/StudentFeaturesTest.php`:**
- Extended `test_take_quiz_page_never_contains_correct_answer()` to assert the take-quiz page now
  includes the fullscreen modal text, `I Understand` button, fullscreen helper/listener code,
  restore re-check, Escape trap, and no `correct_answer` leakage.

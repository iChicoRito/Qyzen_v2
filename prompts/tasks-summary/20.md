# Task 20 - Prevent Warning-Limit Submission and Refresh Bypass

Summary of the implementation for `prompts/tasks-assignment/20.md`: quiz warning counts are now
owned by the server, reaching the configured warning limit immediately finalizes the attempt, and
refreshing or retrying requests cannot reopen the quiz or allow more answers after the limit is
reached.

---

## Plain-language summary (for non-technical readers)

Students can no longer keep answering a quiz by refreshing the page at the moment the final
cheating warning is triggered.

Before this change, the quiz page counted warnings in the browser and tried to submit the quiz
after the last allowed warning. If the student refreshed or interrupted that browser request at the
right time, the automatic submission could be skipped.

Now, each warning is sent to the server right away. The server records the warning count on the
student's quiz attempt. When the warning count reaches the teacher's limit, the server immediately
submits and locks the attempt using the latest answers it has received. After that, draft saves,
stale tabs, reloads, and repeated warning requests cannot change the answers or reopen the quiz.

---

## Technical summary

The implementation reused the existing `tbl_scores` fields: `warning_attempts`, `student_answer`,
`status`, and `submitted_at`. No schema change or warning-event table was added.

### 1. Server-owned warning endpoint

**Fix - `routes/web.php`:**
- Added `POST student/take-quiz/{assessment}/warning`.
- Route name: `student.take-quiz.warning`.
- The route uses the existing `auth`, `verified`, `role:student`, and `throttle:quiz-writes`
  protections from the student quiz route group.

**Fix - `app/Http/Controllers/Student/QuizController.php`:**
- Added `warning(Request $request, Assessment $assessment)`.
- Validates:
  - `reason` as optional text.
  - `answers` as an array.
- Returns the authoritative server warning state:
  - `warnings`
  - `limit`
  - `forced`
  - `redirect_url`
  - `message`
- If the attempt has already been submitted, duplicate or stale requests return the finished
  result redirect instead of creating another score row.

### 2. Atomic warning recording and forced submission

**Fix - `app/Services/QuizGradingService.php`:**
- Added `startAttempt()` so loading the quiz remains the only normal attempt-creation path.
- Changed `saveDraft()` so it updates only an existing `in_progress` attempt and no longer trusts
  the browser's warning value.
- Added `recordWarning()`:
  - locks the active `in_progress` score with `lockForUpdate()`;
  - stores the latest submitted answers;
  - increments `warning_attempts` on the server by exactly one;
  - immediately finalizes the score when `warning_attempts >= assessment.cheating_attempts`;
  - returns an existing finished attempt for duplicate final-warning requests.
- Refactored `grade()` through a shared `finalizeScore()` method so normal submit and
  final-warning submit use the same server-side grading, notification, special-access cleanup, and
  idempotency behavior.

### 3. Draft and reload protection

**Fix - `app/Http/Controllers/Student/QuizController.php`:**
- Draft saves now return `422` when no active `in_progress` attempt exists.
- A final-warning-submitted attempt is treated like any other submitted attempt by the existing
  `take()` gate, so reopening the quiz redirects to the result page.

### 4. Browser warning flow

**Fix - `resources/views/student/take-quiz.blade.php`:**
- Added `warningUrl` for the new server endpoint.
- `bumpWarning(reason)` now sends `reason` and `collectAnswers()` to the server instead of
  incrementing warning state locally.
- The page renders the server-returned warning count.
- When the server responds with `forced: true`, the page disables quiz controls, exits fullscreen
  when needed, and redirects to the server-provided result URL.
- The existing Task 19 fullscreen transition guards remain intact.

### Regression coverage

# Task 22 - Email Code Account Verification

Summary of the implementation for `prompts/tasks-assignment/22.md`: new email-onboarded
accounts now use a six-digit verification code instead of an emailed temporary password, then go
straight to password setup after verification.

---

## Plain-language summary

Before this change, a new user received a temporary password by email and the account link
activated the account immediately. The user then had to sign in with that temporary password
before choosing a new one.

Now the email shows the user's registered email address and a six-digit verification code, without
showing a temporary password. Selecting **Verify Account** opens a secure page with six code boxes.
Typing each digit moves to the next box automatically, and pasting a code fills the boxes.

After the correct code is submitted, the account is verified and the user is taken directly to set
their password. The existing password rules and sign-out-after-change behavior remain in place.

---

## Technical summary

### Email onboarding and code storage

- Added `email_verification_code` and `email_verification_code_expires_at` to `tbl_users`.
- Email onboarding generates a random six-digit code, stores only its hash, and expires it after
  seven days.
- Resending the verification email replaces the previous code without changing an existing password.
- The account-created notification and email now send `verificationCode`, show the registered email,
  and use **Verify Account** as the call to action.
- Offline registration keeps its separate temporary-password export flow and does not send a code.

### Signed verification flow

- The signed `account.activate` link now renders the dedicated verification page instead of
  immediately marking the account verified.
- Its throttled POST endpoint validates only the submitted six-digit code against the user bound by
  the signed URL; the form never accepts an email address or user id.
- Verification locks the user record while consuming the code, rejects inactive, invalid, expired,
  reused, and cross-account codes, then marks the email verified and emits the existing `Verified`
  event.
- A valid verification starts an authenticated session and redirects to `password.force.edit`.

### Password setup and UI

- Added `resources/views/auth/verification-code.blade.php` with six numeric, labelled code fields,
  automatic forward focus, backspace navigation, paste handling, and a **Verify My Account** button.
- The just-verified session may set a new password without entering a current password because no
  temporary password was delivered. This allowance is bound to the authenticated user id and is
  removed after password change.
- Existing flagged users who did not come through the verification flow still must provide their
  current temporary password.

### Tests and verification

- Added focused feature coverage for code delivery/hash storage, code rotation, offline onboarding,
  signed verification success, inactive accounts, expired links/codes, reused codes, cross-account
  codes, the six-field view, and the verified-session password setup path.
- Updated admin and import assertions from temporary-password email expectations to verification-code
  expectations.

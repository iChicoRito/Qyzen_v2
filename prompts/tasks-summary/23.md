# Task 23 - Post-Submission Answer Mismatch Fix + Pass Threshold 75% → 70%

Summary of the implementation for `prompts/tasks-assignment/23.md`: fixed the intermittent bug
where a student's selected answer appeared to differ from what the results page showed after
submission, and lowered the passing score from 75% to 70%.

---

## Plain-language summary

Some students reported that after submitting a quiz, an option they had chosen as (for example)
"A" showed up under a different letter on the results page. It only happened to some students, so
it looked random.

The choices themselves and the grading were always correct — only the **letter label** was wrong.
On the quiz-taking page, when an assessment shuffles its questions, the choice order is reshuffled
on every page load, but each option's letter was being computed from its position on screen instead
of from the choice itself. The results page does not shuffle, so the same choice ended up with a
different letter on each screen. To the student this read as "I picked A but it shows C."

The fix ties each option's letter to the choice's own fixed key (A/B/C/D), so the letter travels
with the content and both the quiz page and the results page always agree — with or without
shuffling. Students who never saw the bug are unaffected.

Separately, the passing score was lowered from 75% to 70%. A score of exactly 70% now shows as
passed everywhere.

---

## Technical summary

### Answer-mismatch root cause and fix

- The choice badge letter was rendered positionally with `chr(65 + $loop->index)` on both the
  take-quiz page and the score-review page.
- The take-quiz page reshuffles choice iteration order on every load when `assessment.is_shuffle`
  is on (Task 21), preserving the `key => text` map but changing display order; the review page
  never shuffles. Positional lettering therefore produced a different letter for the same choice on
  each screen.
- A latent variant existed even without shuffle: bulk-imported questions drop blank choices via
  `array_filter`, so choice keys can be non-contiguous (e.g. A, C, D), and positional lettering
  mislabelled them.
- Grading was never affected — `QuizGradingService` keys strictly by choice key, and the review
  page matches the student's answer by key. Only the visible letter drifted.
- Fix: the badge letter is now the stable choice key itself on both views, since every creation
  path (educator form and bulk import) stores choices under keys A/B/C/D. No controller, service,
  JS, or database change was needed.

### Pass threshold 75% → 70%

- `QuizGradingService::PASS_THRESHOLD` is the single source of truth; changed `0.75` → `0.70`.
- Collapsed two duplicated literals onto the constant instead of hardcoding a new number:
  `OfflineScoreUploadService` now compares against `QuizGradingService::PASS_THRESHOLD` rather than
  a bare `75`.
- Seeder dummy-data literals updated `0.75` → `0.70`.
- Student-facing copy updated on the score-review page and the pre-submit confirmation modal.
- All ≥75% pass-mark references in `CLAUDE.md` and the architecture/roadmap docs updated to ≥70%
  for consistency (the numeric change is a deliberate spec change; the server-side-grading /
  `correct_answer`-never-serialized invariant is unchanged).

### Tests and verification

- Added a boundary test: a 10-question attempt scoring exactly 7/10 (70%) passes, 6/10 (60%) fails.
- Added a render guard: a take-quiz page for a question with non-contiguous keys (A, C, D) labels
  each option by its key ("Choice C", "Choice D", never "Choice B") — this fails on the old
  positional code and proves the mismatch fix.
- Retitled the two former `*75_percent*` grading tests; updated the `DashboardTest` score helper
  from `>= 75` to `>= 70`.
- Confirmed the server-side-grading invariant still holds (`correct_answer` absent from the
  take-quiz payload).
- Verified:
  - `php artisan test` - **330 passed, 1,514 assertions**.
